# Sentry Commit Messages

Follow these conventions when creating commits for Sentry projects.

## Prerequisites

Before committing, always check the current branch:

```bash
git branch --show-current
```

**If you're on `main` or `master`, you MUST create a feature branch first** — unless the user explicitly asked to commit to main. Do not ask the user whether to create a branch; just proceed with branch creation, then re-check the current branch before committing. If still on `main` or `master`, stop — do not commit.

## Commit Procedure (do this every time, in order)

1. Check the branch (above).
2. Write the commit message (Format, Commit Types, Subject/Body rules below).
3. **If the change is user-facing** (see the category table in Changelog), update `CHANGELOG.md` under `## [Unreleased]` — see the Changelog section below. Skip this step only for the excluded types listed there.
4. Stage the code changes **and** `CHANGELOG.md` together, then commit as one.

Step 3 is not optional documentation — it's part of making the commit, not an afterthought.

## Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

The header is required. Scope is optional. All lines must stay under 100 characters.

## Commit Types

| Type | Purpose |
|------|---------|
| `feat` | New feature |
| `fix` | Bug fix |
| `ref` | Refactoring (no behavior change) |
| `perf` | Performance improvement |
| `docs` | Documentation only |
| `test` | Test additions or corrections |
| `build` | Build system or dependencies |
| `ci` | CI configuration |
| `chore` | Maintenance tasks |
| `style` | Code formatting (no logic change) |
| `meta` | Repository metadata |
| `license` | License changes |

## Subject Line Rules

- Use imperative, present tense: "Add feature" not "Added feature"
- Capitalize the first letter
- No period at the end
- Maximum 70 characters

## Body Guidelines

- Explain **what** and **why**, not how
- Use imperative mood and present tense
- Include motivation for the change
- Contrast with previous behavior when relevant
- Use real newlines in commit bodies; never include literal `\n` sequences
- Never include customer data — customer/org names, user emails, support ticket contents, or PII. Describe the technical symptom, not who hit it, and if available, reference the internal ticket (e.g. `Fixes SENTRY-1234`).

## Commit Command Hygiene

When creating commits from the CLI, do not embed escaped newlines like `\n` inside `-m` strings. That produces literal backslash characters in the final commit message.

Prefer one of these patterns:

```bash
git commit -m "type(scope): Subject" \
  -m "First paragraph with real line wrapping.

Second paragraph.

Fixes GH-1234"
```

```bash
git commit
```

Use the editor flow when the message needs careful formatting.

## Footer: Issue References

Reference issues in the footer using these patterns:

```
Fixes GH-1234
Fixes #1234
Fixes SENTRY-1234
Refs LINEAR-ABC-123
```

- `Fixes` closes the issue when merged
- `Refs` links without closing

## AI-Generated Changes

Do not add any AI-authorship marker to commits — no `Co-Authored-By`, no "Generated by AI", no "Written with Claude", or similar. Nothing in the subject, body, or footer should indicate the change was AI-assisted.

## Examples

### Simple fix

```
fix(api): Handle null response in user endpoint

The user API could return null for deleted accounts, causing a crash
in the dashboard. Add null check before accessing user properties.

Fixes SENTRY-5678
```

### Feature with scope

```
feat(alerts): Add Slack thread replies for alert updates

When an alert is updated or resolved, post a reply to the original
Slack thread instead of creating a new message. This keeps related
notifications grouped together.

Refs GH-1234
```

### Refactor

```
ref: Extract common validation logic to shared module

Move duplicate validation code from three endpoints into a shared
validator class. No behavior change.
```

### Breaking change

```
feat(api)!: Remove deprecated v1 endpoints

Remove all v1 API endpoints that were deprecated in version 23.1.
Clients should migrate to v2 endpoints.

BREAKING CHANGE: v1 endpoints no longer available
Fixes SENTRY-9999
```

## Revert Format

```
revert: feat(api): Add new endpoint

This reverts commit abc123def456.

Reason: Caused performance regression in production.
```

## Principles

- Each commit should be a single, stable change
- Commits should be independently reviewable
- The repository should be in a working state after each commit

## Granularity — favor more, smaller commits

Default to splitting work into multiple atomic commits rather than bundling unrelated
changes into one. Maximizing traceability (clean `git bisect`, easy `git revert` of a
single concern, a history that reads as a sequence of intentional steps) matters more
than minimizing commit count.

- If a task touches more than one logical concern (e.g. a fix plus an unrelated refactor,
  or changes to two independent files/features), commit them separately — even within the
  same sitting, even if the user only asked once.
- Commit as soon as a logical unit is complete and the repo is in a working state; don't
  wait to accumulate several unrelated changes into a single "batch" commit.
- Exception: don't fragment a single logical change into artificial pieces just to
  inflate the count — a change that only makes sense as a whole (e.g. a function and its
  only caller, both introduced together) stays in one commit.

## Changelog (CHANGELOG.md)

Every commit with user-facing impact also updates `CHANGELOG.md` at the repo root, staged in the **same commit** as the code change — the changelog should evolve with the code, not get reconstructed later. Format: [Keep a Changelog](https://keepachangelog.com/en/1.0.0/). Versioning: [Semantic Versioning](https://semver.org/).

### Setup (first time in a repo)

If `CHANGELOG.md` doesn't exist, create it:

```markdown
# Changelog

All notable changes to this project are documented in this file.

Format based on [Keep a Changelog](https://keepachangelog.com/en/1.0.0/),
versions follow [Semantic Versioning](https://semver.org/).

## [Unreleased]
```

### Per-commit update

Append one bullet, written from the user's point of view (not implementation detail), under the matching category inside `## [Unreleased]` — create the category subheading if it's missing:

| Commit type | Changelog category |
|---|---|
| `feat` | Added |
| `fix` | Fixed |
| `perf` | Changed |
| `ref` (only if the behavior change is user-visible) | Changed |
| Breaking change (`!` in type or `BREAKING CHANGE:` footer) | Changed, bullet prefixed `**BREAKING:**` |

Skip the changelog entirely for `docs`, `test`, `build`, `ci`, `chore`, `style`, `meta`, `license`, and purely internal `ref` — none of these are perceivable by someone using the project.

### Example (after a few commits)

```markdown
## [Unreleased]

### Added
- Dark mode toggle in settings

### Fixed
- Crash when opening the dashboard with no active session

### Changed
- **BREAKING:** Drop support for config files without a `version` field
```

## Versioning (SemVer)

The version lives wherever the project already tracks it (`package.json`, `pyproject.toml`, `Cargo.toml`, or a bare `VERSION` file) — don't invent a new location, and don't add one to a project that has none.

Do **not** bump the version on every commit — only when the user asks to cut a release (or explicitly says "bump version" / "release"). At that point:

1. Read everything accumulated under `## [Unreleased]` since the last version.
2. Pick the bump level from the highest-severity entry present: any `BREAKING` bullet → major, else any `Added` bullet → minor, else patch.
   - **Exception while the version is `0.x.y`:** the project is pre-1.0 and the public API can still move, so per SemVer, breaking changes bump *minor* (`0.3.0 → 0.4.0`), not major. Only move to `1.0.0` deliberately, when the user says the project is stable/ready.
3. Rename `## [Unreleased]` to `## [x.y.z] - YYYY-MM-DD`, then add a fresh empty `## [Unreleased]` above it.
4. Update the version field in the project's version file to match.
5. Offer to create an annotated tag (`git tag -a vX.Y.Z -m "..."`) — confirm before pushing tags, since that touches shared state.

If the project already has `version-release` injected (CI-driven release trains with auto-tagging and PR-body changelogs), that skill owns releases end to end — don't also maintain a `CHANGELOG.md` file there, to avoid two competing changelogs. This section is for the common case: solo/small repos with no release CI.
